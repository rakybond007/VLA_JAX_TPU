# VLA JAX TPU

This repository is a TPU-first JAX/Flax implementation track for vision-language-action policies:

- keep the VLM backbone and action head separated,
- use a Qwen3-VL-style image/text backbone,
- train a flow-matching Cross-Attn DiT action head,
- support full training on both GPU and TPU,
- support policy serving and evaluation on both GPU and TPU.

## Current Status

The current working path is:

- Pure JAX/Flax VLA policy with the contract `Qwen3-VL backbone -> backbone_features -> flow-matching Cross-Attn DiT action head`.
- Qwen3-VL backbone weights loaded from Hugging Face-compatible checkpoints into the JAX implementation.
- LIBERO LeRobot training pipeline with Qwen formatting, action/state normalization, fixed eval semantics, checkpointing, and websocket serving.
- Full training and evaluation paths for both GPU and TPU execution.
- TPU/GPU timing utilities for measuring one 16-action chunk forward pass.

## Design Notes

The implementation keeps the model boundary intentionally simple:

1. The backbone consumes multimodal observations and emits `backbone_features`.
2. The action head consumes those features plus robot state and predicts an action horizon.
3. Training and inference are thin wrappers around that contract.

The action head used for the main LIBERO experiments is a flow-matching Cross-Attn DiT:

- State and noisy action tokens form the query stream.
- Qwen3-VL hidden states provide the conditioning/memory stream.
- Training predicts flow/velocity targets.
- Inference denoises an action horizon and returns a `(batch, 16, 7)` action chunk for the current LIBERO setup.

## Verified Capabilities

The current implementation has been validated for the core workflows needed by this project:

- Training can run on both local GPU and Google TPU.
- Evaluation can run with either a local GPU-backed policy server or a TPU-backed policy server.
- Checkpoints can be restored across GPU and TPU runs.
- The serving/eval path supports LIBERO rollouts through a websocket policy server and client setup.
- Timing utilities are included for measuring pure model execution of one `(batch, 16, 7)` action chunk.

Detailed private experiment notes can be kept outside the public README, while reusable setup and operation steps are documented here and in `docs/`.

## Setup

```bash
git clone git@github.com:rakybond007/VLA_JAX_TPU.git
cd VLA_JAX_TPU

python -m venv .venv
source .venv/bin/activate
pip install -e .
```

By default, configs save checkpoints under `outputs/checkpoints/...` so the repository does not embed any private cloud bucket. Override this per run with `--checkpoint_dir` when training:

```bash
python -m vla_tpu.training.train \
  --config small_debug \
  --checkpoint_dir outputs/checkpoints/my_smoke_run
```

## TPU Workflow

The main execution path is a TPU VM with the dataset available on the VM and checkpoints written either locally or to GCS.
The TPU scripts look for `.venv-tpu` by default; set `VENV_PATH=/path/to/venv` if your environment name is different.

Set the common paths once:

```bash
export DATASET_ROOT=/mnt/disks/vla-data/datasets/libero_gr00t_delta
export CHECKPOINT_DIR=gs://<CHECKPOINT_BUCKET>/<RUN_NAME>/checkpoints
```

Run LIBERO training on TPU:

```bash
bash run_scripts/train/train_libero_qwen3_vl_tpu.sh
```

The TPU script owns the default model config, global batch, data loader settings, checkpoint cadence, JAX platform selection, and wandb wiring. Override only the values you need:

```bash
NUM_STEPS=30000 \
GLOBAL_BATCH_SIZE=32 \
DATASET_ROOT=/mnt/disks/vla-data/datasets/libero_gr00t_delta \
CHECKPOINT_DIR=gs://<CHECKPOINT_BUCKET>/<RUN_NAME>/checkpoints \
bash run_scripts/train/train_libero_qwen3_vl_tpu.sh
```

Run TPU policy-server evaluation in one shell:

```bash
CHECKPOINT_DIR=gs://<CHECKPOINT_BUCKET>/<RUN_NAME>/checkpoints \
DATASET_ROOT=/mnt/disks/vla-data/datasets/libero_gr00t_delta \
VIDEO_OUT=outputs/libero_rollouts_tpu_axis_flip \
bash scripts/run_libero_eval_tpu_axis_flip.sh
```

Benchmark one 16-action chunk pure model forward on TPU:

```bash
PYTHONPATH=$PWD python -m vla_tpu.tools.benchmark_pure_infer \
  --config libero_full_jax_qwen3_vl_legacy_eval \
  --checkpoint-dir "$CHECKPOINT_DIR" \
  --dataset-root "$DATASET_ROOT" \
  --warmup 5 \
  --iters 30 \
  --use-jit
```

## GPU Workflow

GPU execution is supported with its own entry scripts. Use this path for local development, cluster GPU training, or GPU-backed policy evaluation.
The GPU scripts look for `.venv-local-eval` by default; set `VENV_PATH=/path/to/venv` if your environment name is different.

Set the common paths:

```bash
export DATASET_ROOT=/path/to/lerobot/kimtaey/libero_gr00t_delta
export CHECKPOINT_DIR=outputs/checkpoints/libero_full_jax_qwen3_vl_gpu
```

Run LIBERO training on GPU:

```bash
bash run_scripts/train/train_libero_qwen3_vl_gpu.sh
```

For Slurm clusters that request two GPUs through `sbatch`, this equivalent script is also provided:

```bash
bash run_scripts/train/train_libero_qwen3_vl_2gpu_40k.sh
```

Run GPU policy-server evaluation in one shell:

```bash
bash scripts/run_libero_eval_gpu_axis_flip.sh
```

Benchmark one 16-action chunk pure model forward on GPU:

```bash
PYTHONPATH=$PWD python -m vla_tpu.tools.benchmark_pure_infer \
  --config libero_full_jax_qwen3_vl_legacy_eval \
  --checkpoint-dir "$CHECKPOINT_DIR" \
  --dataset-root "$DATASET_ROOT" \
  --warmup 5 \
  --iters 30 \
  --use-jit
```

## GCP TPU VM Operations

For local GCP work in this repo, use:

```bash
./scripts/gcloudw.sh --version
./scripts/gcloudw.sh auth list
```

Detailed setup notes are in [`docs/gcp_tpu_setup.md`](docs/gcp_tpu_setup.md). The TPU workflow has been exercised with TPU VM shapes such as `v6e-4`; adapt the project, zone, and node names to your own GCP account:

```text
project: <PROJECT_ID>
zone: <ZONE>
node: <TPU_VM_NAME>
accelerator: v6e-4
runtime: v2-alpha-tpuv6e
```

List available accelerator types in a zone:

```bash
./scripts/gcloudw.sh compute tpus accelerator-types list \
  --project=<PROJECT_ID> \
  --zone=<ZONE>
```

Create the TPU VM when it does not already exist:

```bash
./scripts/gcloudw.sh compute tpus tpu-vm create <TPU_VM_NAME> \
  --project=<PROJECT_ID> \
  --zone=<ZONE> \
  --accelerator-type=v6e-4 \
  --version=v2-alpha-tpuv6e
```

For normal reuse, start, check, SSH, and stop the existing VM:

```bash
./scripts/gcloudw.sh compute tpus tpu-vm start <TPU_VM_NAME> \
  --project=<PROJECT_ID> \
  --zone=<ZONE>

./scripts/gcloudw.sh compute tpus tpu-vm describe <TPU_VM_NAME> \
  --project=<PROJECT_ID> \
  --zone=<ZONE>

./scripts/gcloudw.sh compute tpus tpu-vm ssh <TPU_VM_NAME> \
  --project=<PROJECT_ID> \
  --zone=<ZONE>

./scripts/gcloudw.sh compute tpus tpu-vm stop <TPU_VM_NAME> \
  --project=<PROJECT_ID> \
  --zone=<ZONE>
```

Stop the TPU VM whenever it will be idle. A stopped TPU VM preserves the node configuration, but capacity can still be unavailable when starting it again later.

## Artifact Policy

This repository intentionally does not track checkpoints, downloaded datasets, reference repository clones, wandb runs, local GCP credentials, or generated rollout videos outside `docs/assets/`. The main ignored paths are:

- `outputs/`
- `local_checkpoints/`
- `debug_data/`
- `logs/`
- `wandb/`
- `.gcloud/`
- `.venv-*`
- `third_party/*` except `third_party/README.md`
